# code-review

代码评审，遵循 Google 代码评审最佳实践。

## 功能

1. **审查代码变更** - 分析待评审的代码
2. **识别潜在问题** - 发现 bug、安全问题、性能问题
3. **提供改进建议** - 给出具体的改进方案
4. **检查代码规范** - 确保符合团队标准

## Google 代码评审原则

### 评审标准

1. **代码是否正确？** - 功能实现是否正确
2. **代码是否清晰？** - 其他开发者能否理解
3. **代码是否必要？** - 是否过度设计
4. **代码是否安全？** - 是否存在安全隐患
5. **代码是否高效？** - 性能是否合理

### 评审重点

#### 必须关注

- [ ] 代码逻辑正确性
- [ ] 错误处理完整性
- [ ] 安全漏洞（SQL注入、XSS等）
- [ ] 测试覆盖
- [ ] API 兼容性

#### 建议关注

- [ ] 代码可读性
- [ ] 命名规范性
- [ ] 注释完整性
- [ ] 代码复用
- [ ] 性能优化

### 评审反馈原则

#### 好的评审意见

```
建议：这里可以使用 Map 代替 List 查找，时间复杂度从 O(n) 降为 O(1)。

// 示例
const userMap = new Map(users.map(u => [u.id, u]));
const user = userMap.get(userId);
```

#### 不好的评审意见

```
这段代码写得不好。
```

### 评审流程

1. **理解变更目的** - 阅读 PR/CL 描述
2. **检查整体设计** - 架构是否合理
3. **逐行审查代码** - 关注细节
4. **验证测试** - 测试是否充分
5. **提出反馈** - 清晰、具体、建设性

## 评审检查清单

### 功能正确性

- [ ] 代码实现了预期功能
- [ ] 边界条件处理正确
- [ ] 错误处理适当
- [ ] 并发安全（如适用）

### 代码质量

- [ ] 命名清晰有意义
- [ ] 函数/方法长度适中
- [ ] 避免重复代码
- [ ] 遵循 SOLID 原则

### 安全性

- [ ] 输入验证
- [ ] 权限检查
- [ ] 敏感数据处理
- [ ] 依赖安全

### 测试

- [ ] 单元测试覆盖
- [ ] 测试用例有意义
- [ ] 边界条件测试

## 执行步骤

当用户调用此命令时：

1. 获取当前分支的变更（`git diff main`）
2. 分析变更的文件和内容
3. 按照检查清单进行评审
4. 输出评审报告，包含：
   - 发现的问题（按严重程度分类）
   - 改进建议
   - 整体评价

## 评审报告格式

```markdown
## 代码评审报告

### 概述
- 变更文件数：X
- 新增行数：Y
- 删除行数：Z

### 发现的问题

#### 🔴 严重（必须修复）
1. [文件:行号] 问题描述

#### 🟡 警告（建议修复）
1. [文件:行号] 问题描述

#### 🟢 建议（可选优化）
1. [文件:行号] 问题描述

### 改进建议
1. 具体建议内容

### 总体评价
整体评价内容...
```
